<objective>
Push the spectral-phase Raman-suppression optimization from L=30m (-57 dB, established in Phase 12) to L=100m. Ship the numerical infrastructure (longfiber_setup wrapper, checkpointing helper, identity-warm-start from phi@2m) that makes this optimization tractable and the eventual L=200m push in scope. Validate at L=50m first as a cheap stepping stone, then commit the heavy L=100m optimization to burst VM with overnight-safe checkpointing.

Purpose: The PI's publication thread is "long-fiber Raman suppression with preserved phase-shape universality." Phase 12 showed phi@2m retains -57 dB at L=30m (15x opt horizon). This phase asks: can we re-optimize at L=100m starting from phi@2m and do BETTER, and what does the optimal phi(omega) shape look like (pure GVD rescale, or does it carry new structural adaptation)?

Output: Six `scripts/longfiber_*.jl` scripts, ~10 JLD2 files in `results/raman/phase16/`, 5 diagnostic figures, and a FINDINGS.md capturing achieved J_dB, convergence statistics, phi-structure fingerprint, and checkpoint validation.
</objective>

```julia
mutable struct CheckpointBuf
    x_last::Vector{Float64}; f_last::Float64; g_last::Vector{Float64}; iter::Int
end
function longfiber_make_fg!(buf, problem)
    (F, G, x) -> begin
        J, dJ = cost_and_gradient(x, problem)
        buf.x_last .= x; buf.f_last = J; buf.g_last .= dJ; buf.iter += 1
        (G !== nothing) && (G .= dJ); return J
    end
end
function longfiber_checkpoint_cb(buf, out_dir; every=5)
    return state -> begin
        if buf.iter % every == 0 || state.iteration == 0
            @save joinpath(out_dir, "ckpt_iter_$(buf.iter).jld2") x=buf.x_last f=buf.f_last g=buf.g_last iter=buf.iter
            @info "checkpoint" iter=buf.iter f=buf.f_last
        end
        return false
    end
end
```

**What:** Create a self-contained script with `CheckpointBuf`, `longfiber_make_fg!`, `longfiber_checkpoint_cb`, `longfiber_resume_from_ckpt(out_dir)` helpers.

**Why:** L=100m optimization expected 2-4 h wall time. Without checkpointing a mid-run crash loses all progress. (D-F-05)

**How:** Per research §6 code sketch. Config hash: `hash((Nt, time_window, L, P, fiber_id, reltol))` stored in each checkpoint file. Resume helper scans a directory for `ckpt_iter_*.jld2`, picks highest iter, returns `(x, f, g, iter, config_hash)` — caller verifies hash match before resuming.

Additional: wall-clock gate — force a checkpoint if `time() - last_ckpt_time > 600` seconds even if iter stride not hit (guards against multi-minute iterations).

**Verification:** Unit-test in script's `@__FILE__` block: run a trivial Optim L-BFGS on a 10-dim quadratic with checkpoint callback, interrupt after 3 iter (via throwing from callback), resume, confirm final cost matches uninterrupted 5-iter run to 1e-8.

<risks>

- **Burst VM contention with Phase 14:** Phase 14 plan 02 needs burst VM for A/B + robustness. Session F heavy runs must hold /tmp/burst-heavy-lock; check `burst-ssh "ls /tmp/burst-heavy-lock"` before starting Task 5.
