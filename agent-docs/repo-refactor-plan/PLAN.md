# Repo Refactor Plan

This note proposes a serious but non-destructive refactor path for the repository.

The goal is not a rewrite. The goal is to make the project look intentional, preserve the research value that already exists, and reduce the amount of repo surface area a new contributor has to mentally parse before doing useful work.

## Executive summary

The repo already contains a credible package core, real test coverage, and a documented canonical workflow. The main structural problem is that these assets are buried under a large amount of historical and phase-specific material that is still exposed at the top level.

Today the public repo surface suggests:

- too many entry points
- unclear ownership between package code and scripts
- active and historical workflows mixed together
- generated artifacts living next to maintained source docs
- naming that reflects project phases more than stable capabilities

The right next step is to reduce visible chaos without deleting valuable research machinery.

That means:

1. make the canonical workflow unmistakable
2. move legacy and phase-specific work out of the main surface area
3. promote reusable script logic into stable library modules only where the interfaces are already understood
4. archive or hide clutter before deleting anything

## Current pain points

### 1. The public surface is much larger than the real supported surface

The documented happy path is relatively small:

- `make install`
- `make test`
- `make optimize`
- `make sweep`
- `make report`

But the repo exposes a very different picture:

- a large `scripts/` directory with many phase-specific, benchmark, recovery, and one-off drivers
- top-level `presentation-2026-04-17/`, `reports/`, and `results/` directories in the main repo surface
- legacy planning state still present as `.planning/`

This makes it difficult for a new reader to tell what is stable, experimental, historical, or abandoned.

### 2. Too much important behavior lives in include-driven scripts instead of stable modules

There is a real package in `src/`, but much of the practical Raman workflow still depends on script-local code and `include()` chains:

- `scripts/lib/common.jl`
- `scripts/lib/raman_optimization.jl`
- `scripts/canonical/run_sweep.jl`
- `scripts/lib/visualization.jl`
- `scripts/lib/standard_images.jl`

This is workable for research, but it creates maintenance problems:

- weak interface boundaries
- code reuse via file inclusion instead of imports
- harder unit testing of workflow logic
- more accidental coupling between command-line drivers and reusable logic

### 3. Naming is capability-mixed and phase-mixed

The current names combine several incompatible concepts:

- stable capability names: `raman_optimization.jl`, `visualization.jl`
- experimental method names: `continuation.jl`, `acceleration.jl`, `trust_region_*`
- historical phase names: `phase29_*`, `phase31_*`, `phase34_*`
- recovery or cleanup names: `recovery_*`, `regenerate_*`, `fix`, `smoke`

This is a strong sign that the repo accumulated work as a lab notebook rather than being curated as a maintained public codebase.

### 4. Documentation surface mixes maintained docs with generated byproducts

The `docs/` tree currently includes:

- real human-facing docs
- committed LaTeX build byproducts like `.aux`, `.log`, `.out`, `.toc`
- Syncthing conflict files

That makes the docs folder look unmanaged. It also makes it harder to tell which docs are authoritative.

### 5. Results and research artifacts leak into the main product surface

The repo correctly treats `results/` as special operationally, but the top-level tree still exposes:

- `results/`
- `reports/`
- presentation folders

This makes the repository look like a working directory rather than a maintained codebase.

### 6. Package boundaries are real but still underspecified

`src/` already contains a meaningful split:

- `simulation/`
- `analysis/`
- `gain_simulation/`
- `helpers/`

But the package API is still broad and somewhat incidental:

- dict-based parameter passing dominates core interfaces
- script helpers carry domain-specific workflow logic outside the package
- some functionality exists in both package and script layers without a crisp boundary

This is not an argument for rewriting the numerics core. It is an argument for clarifying which layer owns what.

## What must be preserved

Even if parts of the repo look messy, some functionality is clearly valuable and should remain available.

### Preserve as first-class capability

- canonical single-mode Raman optimization workflow
- standard image generation for any run that produces `phi_opt`
- sweep orchestration and report generation
- deterministic environment helpers
- trust-region and continuation machinery that supports current roadmap questions
- reduced-basis tooling from the Phase 31 line of work
- multimode baseline and MMF research workflows
- output-format tooling and result-loading helpers
- regression tests that capture numerics, methodology, and optimizer-surface bugs

### Preserve, but not on the main public surface

- phase-specific reproduction drivers
- one-off benchmark scripts
- recovery scripts for historical result sets
- presentation assembly scripts
- exploratory notebooks

These may still matter for research reproducibility, but they should not read like the repo's primary interface.

## Proposed target structure

The target is a repo where the top-level layout itself communicates intent.

```text
src/
  MultiModeNoise.jl
  simulation/
  analysis/
  gain/
  workflows/           # stable reusable Raman/MMF workflow helpers
  io/                  # output format, manifests, result loaders
  visualization/       # plotting code that is not just a one-off script

scripts/
  canonical/
    optimize_raman.jl
    run_sweep.jl
    regenerate_standard_images.jl
    generate_reports.jl
    validate_results.jl
  research/
    continuation/
    trust_region/
    multimode/
    longfiber/
  archive/
    phase13/
    phase14/
    phase29/
    phase31/
    phase32/
    phase33/
    phase34/

docs/
  README.md
  quickstarts/
  guides/
  physics/
  project/
    recent-phase-synthesis-29-34.md
    why-phase-31-changed-the-roadmap.md
  planning-history/

notebooks/
  exploratory/

results/
  .gitignore-backed runtime outputs only

agent-docs/
  current-agent-context/
  <active internal work folders>
```

The important idea is not the exact folder names. The important idea is the separation:

- package code versus script drivers
- canonical versus research versus archived scripts
- maintained docs versus history
- source versus generated artifacts

## Canonical vs archived vs removed

### Canonical surface

These should be the obvious, supported user-facing workflows:

- `README.md`
- `Makefile`
- `docs/README.md`
- canonical quickstarts
- a small set of stable script entry points
- package-backed visualization and output tooling
- `test/` tiered test entry points

The canonical scripts should eventually shrink to a small set with clear names and stable flags.

### Archive from main surface

These are useful but should stop competing with the canonical interface:

- `scripts/phase*.jl`
- `_phase*` benchmark launchers
- `recovery_*.jl`
- `cost_audit_*` shell wrappers and spawn helpers
- dated presentation folders
- one-off comparison and diagnostic drivers whose value is historical or investigatory

Archiving does not mean deleting. It means moving into an explicit `scripts/archive/` or `docs/planning-history/` context with a short index.

### Remove from tracked main surface

These are real garbage and should leave the main repo promptly once validated:

- LaTeX build byproducts under `docs/`
- Syncthing conflict files under `docs/`
- stale generated report outputs that can be regenerated
- stray shell glue that duplicates a documented canonical command and has no unique logic

These removals are low risk because they do not represent durable scientific capability.

## Safe refactor boundaries now

The following work is low risk and should happen first.

### Phase 1: surface cleanup and categorization

- add one authoritative repo-structure roadmap in `agent-docs/`
- audit top-level directories and document what is source, runtime output, history, or presentation material
- move obvious generated docs byproducts out of version control
- remove Syncthing conflict files from the tracked docs tree
- create a canonical scripts index
- create a legacy scripts index for anything not on the supported path
- rename a small set of user-facing scripts for clarity if aliases or wrappers preserve compatibility

This phase should not change numerics behavior.

### Phase 2: script taxonomy without algorithm changes

- split `scripts/` into `canonical/`, `research/`, and `archive/`
- keep compatibility wrappers at old paths for one release window if needed
- group phase-specific materials under explicit archival subtrees
- isolate burst/ops helpers from scientific drivers
- move presentation generation out of the main script surface

This is mainly file movement, indexing, and naming hygiene.

### Phase 3: promote stable workflow code into package modules

Move only the stable, repeatedly reused workflow logic:

- Raman problem setup helpers
- standard image generation
- result save/load helpers
- manifest writing
- stable validation utilities

Likely destination modules:

- `src/workflows/raman.jl`
- `src/io/results.jl`
- `src/visualization/standard_images.jl`

The goal is that canonical scripts become thin wrappers around package APIs, while experimental scripts can still compose lower-level pieces.

## Medium-term refactor work

These changes are desirable, but should follow the surface cleanup.

### 1. Clarify configuration objects

Today, important interfaces still rely on `Dict{String,Any}` for `sim` and `fiber`.

That flexibility helped research velocity, but it now hurts:

- discoverability
- validation
- type safety
- extension confidence

A safe medium-term move is not a hard replacement. It is introducing typed config wrappers for the stable workflows while continuing to support conversion from dict-based internals.

### 2. Narrow the package API

The package should expose a deliberate workflow API, not just whatever helpers happen to exist.

That means defining supported entry points for:

- setup
- forward solve
- adjoint solve
- cost evaluation
- optimization driver helpers
- result IO
- plotting / standard images

### 3. Separate stable visualization from experimental plotting

Current plotting logic is split between `src/analysis/plotting.jl` and script-local visualization files.

The stable deliverables should live together:

- standard optimization figures
- sweep heatmaps
- report-card style summary figures

Exploratory or phase-specific figures should move behind research or archive boundaries.

### 4. Reshape docs into maintained sections

A professional public repo should let readers answer three questions quickly:

1. how do I run the supported workflow?
2. how is the code organized?
3. where is the historical research record?

The docs tree should be reorganized around those questions rather than file accumulation.

## Work that should wait

These areas are important, but they are not the right first refactor moves.

### 1. Do not rewrite the numerics core around new abstractions yet

The simulation and adjoint kernels are performance-sensitive and numerically delicate. Changing architecture there before cleaning the outer repo surface would create a lot of risk for little visible clarity gain.

### 2. Do not force a full struct migration immediately

Replacing every `sim` and `fiber` dict in one pass would be a large behavioral refactor. Add typed wrappers gradually around stable workflows first.

### 3. Do not delete phase research scripts before indexing their value

Many of the phase-specific files probably encode one-off scientific insight, reproduction steps, or reference implementations. Archive and classify first. Delete only after proving a file is duplicated, stale, or scientifically irrelevant.

### 4. Do not collapse multimode and single-mode workflows prematurely

They share infrastructure but not necessarily the same operational needs. It is better to make that separation explicit than to unify too aggressively and create a blurred abstraction layer.

## Recommended phased execution

### Immediate cleanup

- remove generated docs byproducts and sync-conflict files from tracked docs
- publish a canonical repo map
- classify top-level directories as source, runtime output, or historical material
- define a short supported-script list in docs and README
- mark non-canonical scripts as research or archive candidates

### Medium-term refactor

- reorganize `scripts/` into canonical, research, and archive areas
- move stable workflow helpers from `scripts/lib/common.jl` and related files into package modules
- consolidate result IO and standard-image generation behind stable package APIs
- restructure docs into quickstarts, guides, project notes, and planning history

### Later archival or deletion

- delete duplicated wrappers after compatibility shims are no longer needed
- move old phase-specific reproduction drivers fully into archive
- prune stale results-report glue and dead shell helpers
- remove old files only after each one is either superseded, archived, or confirmed dead

## Practical first milestone

A realistic first milestone for this roadmap is:

1. make the supported workflow obvious
2. make the legacy workflow explicit
3. reduce the top-level embarrassment factor

Concretely, that means:

- a cleaner docs surface
- a smaller visible script surface
- no generated clutter in `docs/`
- no ambiguity about which commands are canonical

If that milestone is done well, later internal refactors become easier because the repo already reads like a maintained system rather than an active scratch directory.
