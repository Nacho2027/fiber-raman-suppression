### Pattern 1: Phase Normalization Before Any Multi-Run Overlay

**What:** Remove global offset and linear group delay from phi_opt before overlaying multiple runs. Without this, physically similar solutions appear unrelated (Pitfall 4 from PITFALLS.md).

**When to use:** Every figure that shows phi_opt or phi_residual from multiple runs. Apply once at data loading time, store normalized arrays.

**Recipe (verified from decompose_phase_polynomial lines 1457–1461):**
```julia
# Source: scripts/visualization.jl:1457-1461, adapted for multi-run use
function normalize_phase(phi_opt, omega_grid, signal_mask)
    phi_signal  = phi_opt[signal_mask]
    omega_signal = omega_grid[signal_mask]
    # Fit a + b·ω (global offset + group delay)
    A = hcat(ones(length(omega_signal)), omega_signal)
    coeffs = A \ phi_signal
    # Subtract from full-length array
    phi_norm = phi_opt .- (coeffs[1] .+ coeffs[2] .* omega_grid)
    return phi_norm, coeffs
end
```

After normalization: `phi_norm` has zero mean and zero linear slope over the signal band. Physically, this removes time-shift and global phase offset degeneracies that the optimizer doesn't distinguish.

**Signal mask:** Use the -40 dB mask from `decompose_phase_polynomial` (already computed on fftshifted `uomega0`). This is the same mask used for the polynomial fit, so fits and normalized phases are consistent.

### Pattern 2: Frequency Axis Construction Per Run

**What:** Each run has its own Nt and time_window, so frequency grids differ. Build the frequency axis from `sim_Dt` (ps) stored in JLD2.

**When to use:** Every figure that plots vs frequency or wavelength.

```julia
# Source: JLD2 data structure (from run_comparison.jl:242 documentation)
# sim_Dt in JLD2 is picoseconds
function build_freq_axis_THz(Nt, sim_Dt_ps, lambda0_nm)
    # Frequency offset from carrier in THz (fftshifted)
    df_THz = fftshift(fftfreq(Nt, 1.0 / sim_Dt_ps))  # THz
    return df_THz
end

function build_lambda_axis_nm(df_THz, lambda0_nm)
    f0_THz = C_NM_THZ / lambda0_nm
    f_abs = f0_THz .+ df_THz          # absolute frequency in THz
    # Only positive frequencies
    pos = f_abs .> 0
    lambda_nm = C_NM_THZ ./ f_abs[pos]
    return lambda_nm, pos
end
```

**Note on sim_omegas:** The JLD2 field `sim_omegas` stores the angular frequency grid (rad/ps or rad/s — check units in raman_optimization.jl). If available, use it directly rather than reconstructing from sim_Dt. From STATE.md: `sim['ωs']` is absolute angular frequency with ω₀ included. The decompose_phase_polynomial function uses `fftfreq(Nt, 1/sim_Dt)` in rad/s (with sim_Dt in seconds) — for plotting purposes, frequency offset in THz is more readable.

### Pattern 3: Polynomial Fit Overlay on Per-Run Detail Panels (Fig 3)

**What:** For each run's detail panel, overlay the raw normalized phase with the GDD+TOD polynomial fit from `decompose_phase_polynomial`. The gap between the two curves is the residual — the 98.9–99.9% unexplained structure.

**When to use:** Fig 3 (per-run detail panels), Fig 7 (residual overlay).

```julia
# Source: reuse decompose_phase_polynomial output fields
result = decompose_phase_polynomial(phi_opt, uomega0, sim_Dt_s, Nt)
# result.gdd_fs2, result.tod_fs3, result.residual_fraction

# Reconstruct the polynomial fit on the full omega grid for overlay:
# phi_poly = gdd * omega^2/2 + tod * omega^3/6  (in rad·s units, then convert)
# The function already returns the fit; to get fit on full grid, recompute:
omega_s = 2π .* fftshift(fftfreq(Nt, 1.0 / sim_Dt_s))  # rad/s, fftshifted
gdd_rad_s2 = result.gdd_fs2 * 1e-30  # fs² → s²
tod_rad_s3 = result.tod_fs3 * 1e-45  # fs³ → s³
phi_poly = gdd_rad_s2 .* omega_s.^2 ./ 2.0 .+ tod_rad_s3 .* omega_s.^3 ./ 6.0
```

Note: `decompose_phase_polynomial` works on fftshifted arrays and normalizes by subtracting linear trend. To overlay on the same plot as `phi_norm`, ensure both are on the same frequency grid (fftshifted) and the polynomial is evaluated on the detrended (normalized) phase coordinate, not the raw phase.

### Pattern 4: Group Delay Computation (Fig 6)

**What:** Group delay = dφ/dω. Shows how the optimizer reshapes arrival time vs frequency — the temporal analog of the phase structure.

**When to use:** Fig 6. This is often more physically intuitive than phase itself because it directly shows "at what time does each frequency arrive."

```julia
# Source: _central_diff already in visualization.jl:172
# omega_grid in rad/THz (convert: 2π·THz)
# group_delay in ps (dφ/dω where ω in rad/THz gives delay in ps)
function compute_group_delay(phi_norm, omega_THz_grid)
    dω = omega_THz_grid[2] - omega_THz_grid[1]  # rad/THz spacing
    gd_ps = _central_diff(phi_norm, dω)          # ps (phase in rad / (rad/THz) = THz⁻¹ = ps)
    return gd_ps
end
```

**Caution:** Group delay is the first derivative of the normalized phase. If `phi_norm` has any remaining linear trend (which it should not after normalization), the group delay will show a nonzero constant. After proper normalization, the group delay should average to zero over the signal band.

### Pattern 5: Before/After Raman Band Comparison (Fig 5)

**What:** For each run, show input spectrum and optimized output spectrum on the same wavelength axis, with the Raman band region zoomed/highlighted. The `uomega0` is the unshaped input field — propagate it forward to get the baseline output, then propagate the shaped input (`uomega0 * exp(i*phi_opt)`) to get the optimized output.

**Critical:** The input field `uomega0` is in FFT order. To get power spectral density: `abs2.(fftshift(uomega0))`. The optimizer cost uses `band_mask` (stored in JLD2) to define the Raman band — use the same mask for the zoom-in overlay.

**Shortcut available:** The before/after spectra were already computed at run time and saved in JLD2 as `J_before` and `J_after` scalars. But for a spectral plot we need the full output field, which requires re-propagation OR reading from the existing `opt.png` figures. **The JLD2 files do NOT store the optimized output field `uωf`** — only phi_opt and uomega0. So Fig 5 requires re-propagating through the NLSE.

**Alternative approach for Fig 5 (no re-propagation):** Use only the input spectrum shape (|uomega0|²) and annotate the Raman band region with the J_before and J_after values. This conveys "how much energy was in the Raman band before and after" without re-running the ODE. The plot would show: input spectrum profile + Raman band shaded + text annotation "J: -X dB → -Y dB (ΔdB = Z)". This is less informative than showing the actual output spectrum but avoids expensive re-propagation.

**Recommended:** Use re-propagation to show the actual shaped output spectrum. Load from JLD2 is sufficient to reconstruct the problem; call `solve_disp_mmf` with `phi_opt` applied. Load the fiber and sim parameters from `common.jl` presets using the `fiber_name` and fiber parameters stored in the JLD2 (betas, gamma, L_m, etc.).

### Anti-Patterns to Avoid

- **Overlaying phases from different runs without normalization:** Pitfall 4 — the offset and linear degeneracy makes identical physics look like random noise.
- **Using the Nt x M shape of phi_opt without squeezing M=1:** phi_opt is stored as shape `(Nt, 1)` (M=1 single mode). All phase arrays should be `phi_opt[:, 1]` before operations.
- **Applying fftshift inconsistently:** `decompose_phase_polynomial` works on fftshifted arrays (line 1443: `fftshift(uomega0[:, 1])`). Either work consistently fftshifted (for plotting) or consistently FFT-order (for mask operations). Don't mix.
- **Plotting sim_omegas (absolute angular freq in rad/ps) directly as a frequency axis:** The result axis would be in thousands of rad/THz which is not interpretable. Convert: `df_THz = (sim_omegas .- mean(sim_omegas)) / (2π)` or reconstruct via `fftshift(fftfreq(Nt, 1/sim_Dt_ps))`.
- **Plotting phi_opt on noise-floor frequency bins:** Apply the signal mask (-40 dB) before plotting. Noise-floor bins have random phase values that will visually dominate the y-axis range and hide the signal-band structure.

## Common Pitfalls

### Pitfall 1: phi_opt Shape — (Nt,) vs (Nt, 1)
**What goes wrong:** JLD2 stores phi_opt with shape matching the optimizer output. In Julia, this may be `(Nt, 1)` (2D array, M=1) or `(Nt,)` (1D vector). Calling `fftshift(phi_opt)` on a 2D array produces a 2D result; indexing `phi_opt[signal_mask]` on a 2D array uses linear indexing which works but may produce unexpected shapes downstream.
**Why it happens:** The optimizer works with shape `(Nt, M)` internally.
**How to avoid:** At load time, normalize: `phi_opt = vec(run["phi_opt"])` or `phi_opt = run["phi_opt"][:, 1]`. Similarly for `uomega0`.
**Warning signs:** `fftshift(phi_opt)` produces a 2D matrix; `phi_shifted[signal_mask]` returns a 2D slice.

### Pitfall 2: sim_Dt Units — Picoseconds in JLD2
**What goes wrong:** `sim_Dt` stored in JLD2 is in picoseconds (documentation in STATE.md: "sim_Dt in JLD2 is picoseconds"). `decompose_phase_polynomial` expects seconds (multiply by 1e-12). Using ps directly gives frequency grid in 1/ps = THz — which is actually correct for THz-unit frequency axes. But if you pass sim_Dt_ps to `decompose_phase_polynomial` unchanged, the polynomial coefficients will be in rad·ps² (not rad·s²) and the GDD conversion factor `* 1e30` (for ps² → fs²) will be wrong by a factor of (1e12)^2 = 1e24.
**How to avoid:** Always convert: `sim_Dt_s = run["sim_Dt"] * 1e-12`. This is already in run_comparison.jl line 242.
**Warning signs:** GDD values of 1e20 fs² or 1e-10 fs² indicate unit mismatch in the frequency grid.

### Pitfall 3: fftshift Consistency Between phi_opt and uomega0
**What goes wrong:** `phi_opt` from the optimizer is in FFT order (not fftshifted). `decompose_phase_polynomial` applies `fftshift` to both `uomega0` (line 1443) and `phi_opt` (line 1453) consistently. When building physics_insight.jl functions, use the same convention throughout.
**How to avoid:** Decide on one convention at the top of the script — "work fftshifted for all plots and masks." Apply `fftshift` once at data load time. Use shifted frequency axes for all plots.
**Warning signs:** The signal band appears at one end of the frequency axis instead of the center; the group delay has a hard discontinuity at the middle of the array.

### Pitfall 4: Runs 2–5 Did Not Converge
**What goes wrong:** Runs 2–5 have `converged=false` in manifest.json. Their phi_opt is the optimizer's best iterate at max_iter, not the true minimum. This does not invalidate the visualization — the phase structure at the last iterate is still physically meaningful — but pattern claims (e.g., "higher N always requires more complex phase") should be stated cautiously.
**How to avoid:** Annotate each figure with convergence status (e.g., open circle for non-converged in scatter plot, "(nc)" in labels). The CONTEXT.md SPECIFICS section notes this is acceptable: we're doing discovery, not making final claims.
**Warning signs:** Phase structure for non-converged runs looks noisy or has oscillatory artifacts without clear physical structure — may reflect optimizer not having settled.

### Pitfall 5: Before/After Comparison Requires Re-Propagation
**What goes wrong:** JLD2 does NOT store the optimized output field. Producing Fig 5 (before/after Raman band comparison) requires running the NLSE forward twice per run (once with phi=0, once with phi_opt). This adds ~30s per run × 5 runs = ~2.5 minutes of compute. If this is not accounted for in the plan's time estimates, the script will appear to hang.
**How to avoid:** Document that Fig 5 has a compute cost. Consider a `skip_reprop=true` flag that skips Fig 5 and generates a placeholder. Alternatively, use only the input spectrum shape for Fig 5 (without the output) — still informative about which runs had strong baseline Raman.
**Warning signs:** Script runs for 5+ minutes — likely stuck in re-propagation. Add `@info` progress logging before each forward solve.

### Pitfall 6: Band Mask Shows Different Frequency Bins Per Run
**What goes wrong:** `band_mask` (stored in JLD2) is a boolean vector of length Nt. Its physical frequency range depends on the grid — `sum(band_mask)` differs across runs. When plotting "Raman band region" as a shaded zone on a frequency axis, the shaded zone should be defined by a physical frequency threshold (~13.2 THz below carrier) not by the mask indices.
**How to avoid:** For figure annotations (shading the Raman band), use `axvspan(-20, -13.2)` (THz offset) applied to each subplot's axes, not the mask indices. The 13.2 THz Raman peak is already encoded as a constant in visualization.jl.
**Warning signs:** Raman band shading covers different fractions of the frequency axis in different subplots.

### Phase Normalization (Reuse of decompose_phase_polynomial internals)
```julia
# Source: scripts/visualization.jl:1441-1480, pattern extracted
function load_normalized_phase(run)
    sim_Dt_s = Float64(run["sim_Dt"]) * 1e-12   # ps → s
    Nt = Int(run["Nt"])
    phi_raw = vec(run["phi_opt"])                 # shape (Nt,)
    u0      = vec(run["uomega0"])                 # shape (Nt,)

    # Build signal mask at -40 dB
    spec_power = abs2.(fftshift(u0))
    P_peak = maximum(spec_power)
    signal_mask = 10 .* log10.(spec_power ./ P_peak .+ 1e-30) .> -40.0

    # Frequency grid in THz (fftshifted)
    df_THz = fftshift(fftfreq(Nt, 1.0 / (sim_Dt_s * 1e12)))  # THz

    # fftshifted phase
    phi_shifted = fftshift(phi_raw)

    # Fit and subtract a + b*omega over signal band
    phi_signal = phi_shifted[signal_mask]
    ω_signal   = 2π .* df_THz[signal_mask]       # rad/THz
    A = hcat(ones(length(ω_signal)), ω_signal)
    coeffs = A \ phi_signal
    phi_norm = phi_shifted .- (coeffs[1] .+ coeffs[2] .* 2π .* df_THz)

    # Zero noise-floor bins for safe unwrapping (BUG-03 fix)
    phi_display = copy(phi_norm)
    phi_display[.!signal_mask] .= 0.0

    return (phi_norm=phi_display, df_THz=df_THz, signal_mask=signal_mask)
end
```

## Open Questions

1. **Does phi_opt have shape (Nt, 1) or (Nt,) in JLD2?**
   - What we know: The optimizer works with (Nt, M) internally; M=1 for SMF runs. JLD2 round-trips native Julia types.
   - What's unclear: Whether `phi_opt` was saved as a matrix or vector.
   - Recommendation: In physics_insight.jl, use `vec(run["phi_opt"])` at load time to normalize shape. Test with `@assert ndims(phi_opt) == 1`.

2. **Does the sim_omegas field in JLD2 contain ω₀ or only the offset frequencies?**
   - What we know: STATE.md says "sim['ωs'] is absolute angular frequency (ω₀ included)". But the JLD2 field name is `sim_omegas` (not `sim["ωs"]`).
   - What's unclear: The exact values stored in `sim_omegas` — are they absolute (rad/s) or offset (rad/s), and in what order (FFT or fftshifted)?
   - Recommendation: Use `sim_Dt` + `fftshift(fftfreq(Nt, 1/sim_Dt_ps))` instead of `sim_omegas` for frequency axis construction. More robust and consistent with `decompose_phase_polynomial`.

3. **Will Fig 5 (before/after) require re-propagation, or is the output field stored?**
   - What we know: JLD2 fields listed in CONTEXT.md canonical refs are: phi_opt, uomega0, convergence_history, J_before, J_after, fiber_name, L_m, P_cont_W, lambda0_nm, fwhm_fs, Nt, sim_Dt, sim_omegas, band_mask. The output field `uωf` is NOT listed.
   - What's unclear: Whether additional fields were saved beyond those listed.
   - Recommendation: Implement Fig 5 using Option A (input spectrum only + J_before/J_after annotation) by default. Add a comment `# TODO: Option B requires re-propagation`.

4. **What will the polynomial residual look like?**
   - What we know: 98.9–99.9% residual means the polynomial fit is nearly useless. The residual has most of the phase variance.
   - What's unclear: Whether the residual has periodic structure (suggesting interference effects), smooth envelopes (suggesting higher-order dispersion), or random-looking structure (suggesting no simple basis).
   - Recommendation: This is exactly what Fig 7 is designed to reveal. No pre-answer needed — let the data speak.
