# Phase 28 — Plan 02

**Title:** Implement the canonical numerical trust-report path  
**Status:** Complete  
**Mode:** Strict GSD execution

## Goal

Turn Phase 28 from a roadmap contract into executable infrastructure on the
canonical single-mode optimization path.

## Scope

- Add a reusable trust-report utility for deterministic environment status,
  boundary fractions, energy drift, gradient-validation summary, and cost-
  surface coherence.
- Persist that report from `scripts/raman_optimization.jl` alongside the
  existing result payload.
- Reuse the same thresholds in `scripts/validation/validate_results.jl`.
- Add focused regression coverage in `test/`.

## Out of Scope

- Amplitude, multivariable, and MMF rollout
- New optimizer methods
- New heavy compute campaigns
- A large rewrite of Phase 18's validation pipeline

## Tasks

1. Create `scripts/numerical_trust.jl` with thresholds, verdict helpers, report
   assembly, and markdown output.
2. Refactor `validate_gradient` in `scripts/raman_optimization.jl` to return
   metrics the trust report can persist.
3. Wire trust-report generation into `run_optimization`, save it into the JLD2
   payload, and emit a markdown sidecar.
4. Point `scripts/validation/validate_results.jl` at the shared thresholds so
   the heavy audit and the live optimizer use the same contract.
5. Add `test/test_phase28_trust_report.jl` covering trust-report construction
   and on-disk persistence from `run_optimization`.

## Verification

- `julia --project=. test/test_phase28_trust_report.jl`
- Re-run `julia --project=. test/test_phase27_numerics_regressions.jl` to ensure
  Phase 28 does not regress the numerics-audit bugfixes.

## Success Criteria

- Trust-report fields are generated from live optimizer data, not hand-written.
- The canonical optimizer writes a machine-readable trust report plus a
  human-readable markdown summary.
- The Phase 18 validator reuses the same trust thresholds.
- The new regression file passes locally.
