Create `scripts/benchmark_threading.jl` â€” a self-contained benchmark that tests all
threading opportunities in the codebase. The script MUST NOT change Nt, ODE tolerances,
or solver choice.

The script structure:

1. **Setup** â€” Load MultiModeNoise, include common.jl and raman_optimization.jl.
   Use `setup_raman_problem` with default SMF-28 parameters (L=1.0m, P=0.05W, Nt=2^13).
   Report `Threads.nthreads()` and `BLAS.get_num_threads()` at startup.

2. **Benchmark A: FFTW Threading** â€”
   FFTW supports internal threading via `FFTW.set_num_threads(n)`. This controls how
   many threads each FFT call uses internally (independent of Julia threads).
   - Create FFTW plans with `FFTW.MEASURE` at the working Nt for both (Nt,1) and (Nt,1,1) shapes
   - For n_fftw_threads in [1, 2, 4, 8]: call `FFTW.set_num_threads(n)`, create fresh plans,
     time 100 forward+inverse FFT pairs on (Nt, M) arrays
   - Also time a full `cost_and_gradient` call with each FFTW thread count
     (since FFTs are called many times inside the ODE solver per step)
   - Report: FFT-only time and full forward-adjoint time for each thread count

3. **Benchmark B: Tullio/LoopVectorization Threading** â€”
   Tullio uses LoopVectorization internally and respects `Threads.nthreads()`.
   For M=1 (single-mode), the Tullio contractions collapse to scalar operations on
   the t-dimension (shape Nt x 1 x 1 tensors), so threading benefit is in the loop
   over Nt points.
   - Time the core `@tullio` operations in isolation by extracting the Kerr contraction
     pattern: `delta[t,1,1] = gamma[1,1,1,1] * (v[t,1]*v[t,1] + w[t,1]*w[t,1])` and the
     Raman contraction. Use `@tullio` with `threads=false` and `threads=true` keywords.
   - Time a full `cost_and_gradient` call (which exercises all Tullio contractions inside
     the ODE RHS, called hundreds of times per solve)
   - NOTE: Tullio threading requires Julia to be started with `-t N`. The script should
     detect the current thread count and report what it is. If nthreads()==1, note that
     Tullio threading is unavailable and report the single-thread baseline only.

4. **Benchmark C: Multi-Start Optimization Parallelism** â€”
   This is already implemented in `benchmark_optimization.jl` via `Threads.@threads`.
   Replicate a smaller version: run `n_starts=4` optimization runs (max_iter=10 each)
   both sequentially and (if nthreads()>1) with `Threads.@threads`.
   - Sequential: loop over 4 starts, deepcopy fiber, call `optimize_spectral_phase`
   - Threaded: use `Threads.@threads` with deepcopy per thread (same pattern as
     `multistart_optimization` in benchmark_optimization.jl)
   - Report wall time for sequential vs threaded

5. **Benchmark D: Embarrassingly Parallel Forward Solves** â€”
   When running parameter sweeps or gradient validation, independent forward propagations
   can run in parallel. Benchmark: run 8 independent `cost_and_gradient` calls with
   different random phases.
   - Sequential: simple loop
   - Threaded: `Threads.@threads` with per-thread deepcopy of fiber
   - Report wall time for sequential vs threaded

6. **Summary Table** â€”
   Print a box-drawing summary table (matching codebase convention) with columns:
   | Opportunity | 1 thread | N threads | Speedup | Notes |
   Include a note for each about whether this is "free" (just a flag) vs requires code changes.

- Run each benchmark 3 times and take the median to reduce noise.
- JIT warmup: call each function once before timing.
- The script should be runnable as `julia -t 8 scripts/benchmark_threading.jl`.
